# Spec: 卡片总览与复习洞察仪表盘

## Background

Thinklish 已支持从学习日志生成 Anki 风格卡片，并在独立的「复习」流程中处理到期卡片。用户当前缺少**集合层面**的可见性：共有多少张卡、哪些处于待复习、哪些仍在巩固、哪些已趋于稳定。侧边栏虽有复习相关提示，但无法一眼理解整体学习资产与 SRS 分布。

本规格在**不改动既有复习交互契约**的前提下，增加以卡片为中心的**总览入口**与**洞察仪表盘**，帮助用户建立对词汇卡库存与进度的全局心智模型，并降低「不知道自己有多少卡、卡在哪里」的焦虑。

## Goals

1. 提供**显式、可发现**的卡片总览入口，与「文章 / 学习日志 / 复习」并列或逻辑等价地进入用户主路径。
2. 展示**可解释的聚合指标**：卡片总量、与复习节奏相关的分布（至少覆盖「待复习 / 复习中（含短间隔巩固）/ 已掌握（长期稳定）」三类用户可理解语义）。
3. 提供**洞察型仪表盘**：用分布与趋势感知的呈现方式，让用户快速判断「今天要压多少量」「长期是否在积累掌握」。
4. 支持**可筛选的卡片清单**，便于从总览下钻到具体卡片，并与现有复习、日志生成卡片等流程**语义一致**（同一套数字与状态含义）。
5. 与现有**复习入口上的数量提示**在含义上对齐，避免同一应用内出现互相矛盾的「待复习」数字。

## Non-Goals

1. **不定义或替换 SRS 算法**（间隔、易度因子计算规则保持产品现状；本规格只要求**展示与分组**与现有数据语义一致）。
2. **不要求**本期实现卡片批量编辑、批量删除、模板化牌面、或 Anki  deck 同步。
3. **不要求**将「学习日志上的 mastery_status」与「卡片 SRS 状态」强制合并为单一字段；若两者并存，规格仅要求**界面层**对用户说明清晰、不误导。
4. **不要求**导出格式或 IPC 对外契约的破坏性变更。
5. **不要求**移动端或 Web 版适配（桌面 Electron 体验即可）。

## Feature List

特性按**依赖与价值**排序：先解决「找得到、有数据定义」，再堆叠「仪表盘与清单」，最后做「全应用数字一致」。

---

### F1: 卡片总览主导航入口与页面骨架

**What:** 在应用主导航中增加（或等效提升）一个指向「卡片总览 / 卡片仪表盘」的入口，并承载该功能的顶层页面骨架（含标题、简短说明、空状态与加载预期）。

**Why:** 没有稳定入口，后续统计与图表无法被用户发现；先固定信息架构有利于后续 sprint 独立交付可视化与列表。

**Acceptance Criteria:**

- AC-1.1: 首次启动或常规使用流程中，用户**无需记忆隐藏手势**即可从主导航进入卡片总览区域。
- AC-1.2: 当用户尚无任何卡片时，总览页展示**明确的空状态**（说明如何在学习日志中生成卡片），且不出现误导性的「全零图表」或错误提示。
- AC-1.3: 当数据尚未就绪时，总览页呈现**可理解的加载或占位状态**，且不会出现与「无卡片」相混淆的文案。
- AC-1.4: 从总览页用户可**返回**已存在的核心区域（文章 / 日志 / 复习）而不被困在单层路由中。

**Priority:** P0  
**Dependencies:** 无（仅依赖现有应用壳与导航模式）

---

### F2: 卡片聚合指标与状态分桶（产品语义）

**What:** 定义并交付一组**用户可见**的卡片统计：至少包括「总卡片数」以及三类分布——**待复习**、**复习中（巩固）**、**已掌握（长期稳定）**——其含义与现有卡片字段（如下次复习时间、重复次数、间隔等）**可追溯、可验收**；并在总览体验中展示这些数字。

**Why:** 仪表盘的价值取决于指标是否可信、是否与复习行为一致；分桶规则必须在产品层说清楚，避免工程与 UX 各自解释。

**Acceptance Criteria:**

- AC-2.1: 总览区域展示**总卡片数**，且该数字与用户通过「可从现有能力枚举的全部卡片」在相同数据快照下**一致**（允许合理延迟，但需在验收场景下可复现一致）。
- AC-2.2: 展示**待复习**数量，其定义与复习列表中「当前应出现」的卡片集合**兼容**（用户能理解：待复习数字与去复习的体验指向同一类卡片）。
- AC-2.3: 展示**复习中（巩固）**数量：表示尚未进入长期稳定节奏、仍在短周期巩固的卡片子集；边界情况（如新卡、从未复习）有**文档式简短说明**或图例，避免与「待复习」完全混同。
- AC-2.4: 展示**已掌握（长期稳定）**数量：表示已进入较长间隔或高稳定度的子集；与「待复习」互斥展示逻辑在用户视角**可理解**（同一张卡在任一瞬间只属于一个主分类）。
- AC-2.5: 在用户完成一次复习或从日志**新生成一张卡片**后，返回总览时指标**反映最新状态**（同一会话内可验证，无需重启应用）。

**Priority:** P0  
**Dependencies:** F1（展示位置）；依赖现有卡片数据模型与复习行为，但不规定实现路径

---

### F3: 洞察仪表盘可视化

**What:** 在总览页用**图形化摘要**呈现 F2 的分桶结果（例如分布图、条形对比、环形进度等任一或组合），并辅以关键 KPI（如「今日待复习」「近 7 日复习量」类指标中至少一项，若数据可得）。

**Why:** 数字列表认知负荷高；可视化帮助用户快速判断压力与长期趋势。

**Acceptance Criteria:**

- AC-3.1: 用户**无需滚动首屏之外**即可看到总卡片与三类分布的**至少一种**图形化摘要（与 F2 数字同源）。
- AC-3.2: 图形与数字**一致**：任一展示形式上的份额与旁边（或提示中）的数值**匹配**，不出现图表与表格矛盾。
- AC-3.3: 提供**图例或简短说明**，解释各颜色/分区对应「待复习 / 复习中 / 已掌握」的用户语言（可与 F2 说明复用）。
- AC-3.4: 在**暗色与亮色**主题下（若应用支持主题切换），仪表盘**可读性**保持可用（对比度足够、文字不淹没）。
- AC-3.5: 在**大量卡片**（验收用例中定义的下限，例如不少于约定张数）下，仪表盘仍在**可接受时间内**完成呈现，且不阻塞页面其余部分的基本交互。

**Priority:** P1  
**Dependencies:** F2

---

### F4: 可筛选的全量卡片清单

**What:** 在总览区域内提供**浏览全部卡片**的列表视图，支持按 F2 定义的状态分桶筛选，并展示足以辨认卡片的内容预览（如正面文本摘要）及与复习相关的下一复习时间或状态标签。

**Why:** 用户需要从大数统计下钻到「具体是哪些词卡住了」，并验证仪表盘与自身记忆一致。

**Acceptance Criteria:**

- AC-4.1: 列表默认展示**用户库中全部卡片**或等价的全集入口（例如默认「全部」且条数与 F2 总数一致）。
- AC-4.2: 用户可按**待复习 / 复习中 / 已掌握**（与 F2 一致）筛选，筛选后条数与 F2 对应分桶数字**一致**。
- AC-4.3: 每条列表项展示**可识别的正面内容预览**；过长文本以省略或折叠方式处理，不破坏布局。
- AC-4.4: 从列表可到达**现有复习或卡片相关能力**中的至少一种（例如跳转复习队列、打开相关日志上下文、或进入卡片详情），且路径清晰。
- AC-4.5: 空筛选结果时展示**明确空状态**（非错误），并提示如何调整筛选或去生成卡片。

**Priority:** P1  
**Dependencies:** F2（分桶定义）；F1（页面承载）

---

### F5: 与侧边栏复习提示的跨界面一致性

**What:** 规范并落实「侧边栏（或主导航）上已存在的复习相关数量提示」与总览页「待复习」指标在**语义与时间范围**上的一致策略；若因产品原因必须不同（例如「仅今日到期」vs「全部逾期」），则须在 UI 上以**简短标签**区分，避免用户认为应用出错。

**Why:** 多入口同一指标不一致会摧毁信任；本特性是体验收尾与风险收敛。

**Acceptance Criteria:**

- AC-5.1: 在默认设置下，侧边栏复习提示与总览「待复习」**数值一致**，或**明确标注统计口径差异**（用户能读懂为何不同）。
- AC-5.2: 完成复习使队列减少后，侧边栏与总览的待复习数字**同步更新**（同一会话可验证）。
- AC-5.3: 新生成卡片若立即进入待复习集合，则两侧**同时体现**该增量（或同时不体现，但需与产品规则一致且可见说明）。
- AC-5.4: 无卡片时，侧边栏与总览均**不出现**误导性的非零待复习提示。

**Priority:** P1  
**Dependencies:** F2；依赖现有复习提示行为

---

## Risks

1. **分桶语义与底层字段映射争议：** `next_review_at`、`interval`、`repetitions` 等与用户语言「复习中 / 已掌握」不完全同构，需在 F2 明确产品定义，否则仪表盘与列表会系统性「感觉不对」。
2. **与 lookup `mastery_status` 的混淆：** 用户可能以为日志里的「掌握」等于卡片「已掌握」；若不做解释，会引发支持成本与错误操作。
3. **大数据量性能：** 全量列表 + 聚合在卡片数极大时可能拖慢首屏；需在验收中约定合理上限或渐进加载期望。
4. **指标延迟与一致感：** 若采用缓存或异步聚合，可能出现短暂数字不一致；需在产品上接受或定义刷新策略。
5. **导航项膨胀：** 新增入口可能稀释专注度；需依赖 F1 的清晰命名与 F5 的数字一致来降低认知成本。

## Dependencies

- 现有 **Electron 桌面应用**导航与页面结构。
- 现有 **卡片数据模型**与复习流程（到期筛选、复习提交、从日志生成卡片）。
- 现有 **学习日志**作为卡片来源的用户路径。
- **设计系统**（颜色、排版、空状态模式）以保障仪表盘在不同主题下的可读性。

## Suggested Sprint Order

1. **Sprint A — F1:** 入口与页面骨架（可含空状态与导航文案定稿）。  
2. **Sprint B — F2:** 聚合指标与分桶语义落地，并在总览展示数字。  
3. **Sprint C — F3:** 洞察可视化与 KPI 扩展。  
4. **Sprint D — F4:** 全量列表与筛选。  
5. **Sprint E — F5:** 与侧边栏复习提示的对齐与口径治理。
